# 投稿包与格式化

## 投稿包与格式化

---

**适用场景**

稿件正文已在 Markdown 中完成主要修改，主干判断层工作已基本结束，当前任务以 **投稿包执行性落地** 为主，包括但不限于：

- 目标期刊格式要求逐项核对
- Markdown → Word（`.docx`）转换
- 三线表构建与验证
- 参考文献格式化与完整性核查
- 声明章节补齐
- 图表、补充材料与投稿物料整理
- Cover Letter 一致性检查
- Checklist、代码仓库 URL / License 等投稿包信息回填
- 最终提交前验证

本 prompt 的目标不是重开主线诊断，而是把已经基本稳定的稿件，整理成 **可提交的投稿包**。

---

**进入与退出规则**

适合进入：
- 主文修改已基本完成，当前任务以投稿包、Word、图表文件、声明、checklist、代码仓库 URL / License 回填等执行性落地为主
- 已完成主干判断层工作，当前重点不再是主线诊断，而是提交物料整理
- 当前问题主要集中在格式、物料、引用、声明、Word、图表命名、cover letter 一致性等层面

默认退出：
- 若执行过程中未发现主干硬伤，则继续留在本 prompt 内完成投稿包生成与最终核查
- 若已完成全部投稿物料与格式化任务，则以"是否达到可提交状态"作为本 prompt 的结束判断

默认回退：
- 结论边界或写强问题 → 回到 `academic-revision-system / 第 B6 步`
- 方法结果一致性问题 → 回到 `academic-revision-system / 第 B7 步`
- 统计与偏倚表达问题 → 回到 `academic-revision-system / 第 B8 步`
- 报告规范、引文或声明映射问题 → 回到 `academic-revision-system / 第 B9 步`
- 图表、补充材料或承载闭环问题 → 回到 `academic-revision-system / 第 B10 步`
- 整包一致性与投稿前终检问题 → 回到 `academic-revision-system / 第 B11 步`
- 修回稿 response letter—正文一致性问题 → 回到 `academic-revision-system / 第 B13 步`

不要现在做：
- 不在本 prompt 内重开主线诊断
- 不在本 prompt 内新增复杂分析或重做核心统计
- 不把本 prompt 当作主干修稿入口使用
- 不用"格式整理"替代应回主干处理的问题

---

**基本规则**

请遵守以下规则：

1. 先查期刊要求，再做转换，不要用默认格式生成后再改
2. 不虚构参考文献，不补造未提供的期刊要求
3. 不替用户填写作者信息，用 `[USER: INSERT ...]` 占位
4. 不借"格式整理"之名重写研究结论；若发现真实硬伤，只标红说明
5. 三线表是医学期刊硬标准，不接受全边框表格
6. 所有数字必须在正文、表格、图题、补充材料之间一致
7. 构建产物必须清理，不提交 `node_modules`、`package.json` 等临时文件
8. **如果缺少目标期刊官方 Author Guidelines 原文，必须立即停机并明确请求用户手动获取（PDF / 截图 / sample manuscript）。在用户提供官方原文之前，不得：**
   - 用搜索摘要、二手整理、第三方引用管理器代替
   - 用其他期刊的格式要求推断
   - 跳过 Phase 1 直接进入 Phase 2
   - 凭"医学期刊通常如何"做经验推断
9. 如稿件或投稿包中存在 `代码仓库 URL` / `License` 占位，且用户明确允许使用 GitHub MCP / skills，可在不扩大公开范围的前提下，完成最小必要的代码公开准备（如确认仓库、补齐 `LICENSE`、推送最小可复现代码子集），并将最终仓库 URL 与 License 名称回填到相应占位处；若当前条件不足以安全完成，则保留占位并明确缺口
10. **Phase 0 是 Phase 1 的前置硬阻塞**；Phase 1 只有在已获取目标期刊官方要求原文并完成关键格式项核对后，才能进入 Phase 2；否则必须停机
11. 多来源信息冲突时，必须显式标记 `⚠️ 信息源冲突，待确认`，不得选择"更方便实现的那个"继续
12. 允许使用 `docx`、`python-docx`、`pandoc` 等工具生成 `.docx`，但必须通过统一验证脚本后才能声明完成
13. 最终验证必须细化到字段级，不接受"整体看起来合规"的判断
14. **每次重大修改前必须备份**；不允许"东补西削"反复修改导致信息丢失（详见 Phase 2 安全规则）

---

**执行流程（按顺序，不可跳步）**

---

**Phase 0：用户阻塞项前置确认**

在 Phase 1 启动前，必须先向用户确认以下"硬阻塞项"。任一项缺失则停机，不得进入 Phase 1。

| 类别 | 阻塞项 | 缺失后果 |
|------|--------|---------|
| 期刊定位 | 目标期刊名称（最好提供 1 主选 + 1 备选） | 无法启动 Phase 1 |
| 期刊定位 | 文章类型（Original Investigation / Letter / Review / etc.） | 字数限制、结构、摘要要求都不同 |
| 作者信息 | 通讯作者姓名 + 邮箱（其他作者可后补但通讯必填） | DOCX 必填字段，投稿系统硬验证 |
| 伦理 | IRB / Ethics 批件号或 waiver 文件号 | 期刊投稿系统硬验证项 |
| 资助 | Funding 状态（具体来源 / 无 / 仅机构内） | 多数期刊要求精确措辞，不接受"None" |
| 数据政策 | 是否允许公开本院数据 / 仓库 URL | 影响 Data Availability 措辞 |
| AI 使用 | AI 工具使用范围 + 是否同时在 cover letter 声明 | 期刊政策不同 |
| 二手引用 | 任何"unpublished"或"in preparation"参考是否已投稿 / 有 preprint | 影响 Reference 格式 |

输出"Phase 0 阻塞清单"，列出每项的状态：
- ✅ 已确认（附用户提供的内容）
- ❌ 缺失（请求用户提供）
- ⚠️ 部分确认（说明仍需补全什么）

任一项 ❌ 即停机。所有 ✅ 后才进入 Phase 1。

---

**Phase 1：目标期刊格式要求采集**

在做任何转换之前，先查阅目标期刊的 Author Guidelines（通常在期刊官网的 "Guide for Authors" 或 "Instructions for Authors" 页面），提取以下全部要求：

| 维度 | 需要确认的具体项 |
|------|-----------------|
| 摘要 | 字数上限、结构格式（是否要求 Background/Methods/Results/Conclusions 等） |
| 正文 | 字数上限、是否包含 References/Tables/Figure legends |
| 参考文献 | 数量上限、格式风格（Vancouver/Harvard/数字上标等）、标点规则、DOI 要求 |
| 表格+图表 | 合计数量上限、主文与补充材料如何计数 |
| 关键词 | 数量要求、排序规则、拼写规范（UK/US English） |
| 必需章节 | Authors' contributions、Acknowledgements、Declaration of interests、AI declaration、Funding、Data availability、Ethics statement 等 |
| 章节顺序 | 期刊规定的各章节排列顺序 |
| 标题页 | 作者姓名格式、通讯作者标注方式、Running title 字符上限 |
| 页码 | 位置（页眉/页脚、左/右/居中）、起始页 |
| 格式 | 字体、字号、行距、页边距、页面尺寸（A4/Letter） |
| 统计报告 | *P* 值格式、CI 格式、效应量+CI 是否强制 |
| 缩写规则 | 摘要中是否限制缩写、是否需要缩写表 |
| 文件格式 | Word / PDF / LaTeX / 其他 |
| 补充材料 | 命名规范（如 SDC / eFigure / Supplementary Table）、是否独立文件、是否需要单独标题页 |
| Cover Letter / Checklist | 是否有模板、是否有强制 checklist |

**Phase 1 采集顺序优先级：**

1. **PDF 优先**：必须直接访问期刊官网下载 Author Guidelines PDF，而非 HTML
2. **HTML 仅作为补充验证**：不得作为主要来源
3. **若官网仅提供 HTML（无 PDF），必须截图保存关键格式说明**
4. **关键样例（reference format example、title page template）必须从 PDF / 官网原文逐字提取**，不允许"按 Vancouver 通用规则推断"

输出要求：
- 列出一张"**期刊要求 vs 当前稿件状态**"对照表
- 标明：
  - 当前状态：已合规 / 不合规 / 未知
  - 证据来源：官网 PDF / 官网 HTML / 用户提供 / 二手来源
  - 需要改动：简要说明
  - 是否阻塞后续流程：是 / 否

---

**Phase 1 停机判据（必须同时满足才能进入 Phase 2）**

1. 已直接访问目标期刊官方 Author Guidelines 原文（**官网 PDF 优先；否则官网 HTML + 用户提供截图**），而非仅依赖搜索摘要、第三方整理或引用管理器
2. 已获取以下关键信息的 primary source：
   - 摘要字数与结构
   - 正文字数
   - 参考文献格式（**含真实示例**）
   - 图表/表格数量上限
   - 必需声明章节
   - 章节顺序
   - 页面与字体格式
   - 引用样式与引用位置（如标点前/后、方括号/上标）
3. 若任一项只能依赖二手来源，必须在输出中标记：`⚠️ 基于二手来源，需作者确认`
4. 不得使用"大多数期刊通常……"之类的经验推断替代官网原文

---

**Phase 1 强制规则：无法直接获取 Guideline 时的处理**

如果出现以下任一情况，**必须立即停机并请求用户手动提供**：

1. WebFetch 返回 401 / 402 / 403 / 404 / 5xx
2. 期刊官网在 paywall / Cloudflare / 登录墙后
3. PDF 链接失效或返回 HTML 错误页
4. WebSearch 仅返回二手来源或第三方整理
5. 同一信息在不同来源间冲突且无法判定权威

停机时必须输出：

```text
⚠️ Phase 1 阻塞：无法直接获取目标期刊 [期刊名] 的官方 Author Guidelines

已尝试的获取方式：
- [ ] 期刊官网 HTML：[结果]
- [ ] 期刊官网 PDF：[结果]
- [ ] 出版商投稿系统说明：[结果]
- [ ] PMC / 公开镜像：[结果]

仍缺失的关键信息：
- [ ] 摘要字数与结构
- [ ] 正文字数限制
- [ ] 参考文献格式（含真实示例）
- [ ] 必需声明章节
- [ ] 页面与字体格式
- [ ] 引用样式（上标 / 方括号 / 标点位置）
- [ ] 表格/图表数量上限

请用户提供以下任一项后再继续：
1. 直接下载的官方 Author Guidelines PDF（最优）
2. 期刊官网截图（覆盖关键格式说明部分）
3. 已发表文章的样例 PDF（用于格式参考）
4. 目标期刊近期 sample manuscript（如有）
```

收到用户提供的材料后才能继续 Phase 1。**禁止**：
- 凭"医学期刊通常如何"经验推断
- 用其他类似期刊的要求替代
- 跳过 Phase 1 直接进入 Phase 2
- 用 Phase 1.5（已发表文章对照）来推断 Guidelines 的格式要求

---

**信息冲突处理规则**

多来源信息冲突时，优先级如下：

1. 官网 PDF
2. 官网 HTML
3. 出版商提供的官方模板
4. 投稿系统内的官方说明
5. 第三方引用管理器 / 搜索摘要 / 博客整理

若冲突仍未解决：
- 必须标记 `⚠️ 信息源冲突，待确认`
- 必须列出冲突点
- 不得选择"更方便实现的那个"继续

---

**Phase 1.5：对照目标期刊已发表文章**

在完成官方要求采集后，再读取目标期刊最近 2–3 篇 Original Article 的正式发表版本，用于核对以下格式细节：

- 引用位置（标点前/后）
- 参考文献格式与缩写
- 图注格式
- 表格样式
- 标题页与声明章节顺序
- 页码与 running title 呈现方式
- 拼写规范（UK/US English）
- 是否有期刊实际执行与官网文字略有差异的地方

**Phase 1.5 替代方案优先级**

paywall 等情况下的替代方案，从优到劣：

1. **用户提供已发表文章 PDF**（最佳）
2. **PMC 上同期刊免费全文**（如有）
3. **期刊提供的 sample manuscript 模板**（多数期刊有）
4. **期刊作者指南中的 reference example**（最低保证）
5. **同出版社（如 Wolters Kluwer/LWW）其他期刊文章**（最后兜底，需标注 ⚠️）

若全部失败：明确停机，请求用户提供。**不允许凭经验推断**。

输出要求：
- 列出"**官方指南 vs 已发表文章**"差异表
- 若二者不一致，必须标记：`⚠️ 指南与已发表文章存在差异`
- 不得静默按其中一方执行
- 默认以官网 Guidelines 为主；若已发表文章呈现稳定且一致的不同做法，须在输出中提示用户确认

---

**Phase 2：Markdown 稿件格式预处理**

在转换为 Word 之前，先在 Markdown 源文件中完成以下修改。

**Phase 2 修改优先级**

**P0 - 内容硬阻塞（先做）：**
- Word count 削减/扩展到符合限制
- 章节重命名（如 "Methods" → "Materials and Methods"）
- 章节顺序调整
- 必需子节添加（如 "Statistical Analysis" at end of M&M）
- 必需声明添加（Funding / AI / Sex variable 等）

**P1 - 格式硬阻塞：**
- 参考文献格式（colon vs period、en-dash、journal name case）
- 引用样式（superscript vs bracket、before/after punctuation）
- DOI 补齐
- 缩写首次定义

**P2 - 渲染层（在 Phase 3 处理，不要在 Markdown 阶段做）：**
- 字体、字号、行距、对齐
- 三线表
- 页边距、页码、页眉

注意：P2 项目不能在 Phase 2 处理，因为 Markdown 不承载这些信息。

**Phase 2 修改安全规则**

每次重大修改（>50 词增删 / 章节重排 / 参考文献格式批量改动）前：

1. 备份当前 Markdown 到 `manuscript_v[N]_backup_[timestamp].md`
2. 修改后用 diff 工具核对：
   - 数字是否被误改
   - 关键卖点措辞是否丢失
   - 边界限定语是否被一并删除
3. 若修改导致 word count 反复横跳（如 Discussion 削减 → 新增子节 → 再削减），**必须回退到上一稳定版后重新规划**，不允许"东补西削"

---

**2.1 章节顺序调整**

按期刊要求的顺序重排所有章节。常见医学期刊顺序例如：

```text
Title page → Abstract + Keywords → Introduction → Methods → Results →
Discussion → Conclusions → Authors' contributions → Acknowledgements →
Declaration of interests → AI declaration → Funding → References →
Tables (含标题和脚注) → Figure legends
```

若目标期刊要求与此不同，以期刊要求为准。

---

**2.2 参考文献**

- 核查编号是否按正文首次出现顺序连续排列
- 核查是否所有文献都在正文中被引用，无孤立项
- 核查数量是否在期刊上限内
- 核查是否存在明显缺口（领域关键文献未引），按需补充
- 按字段级别逐项核对参考文献格式，不得只做笼统判断

必须逐项确认以下内容：

- 编号顺序：是否按首次出现排序
- 引用样式：方括号 / 上标 / 其他
- 引用位置：标点前 / 标点后
- 空格规则：引用前后是否留空格
- 期刊名：全称 / NLM 缩写 / 其他
- 期刊名字体：正体 / 斜体 / 小型大写（如 Anesthesiology 要求）
- 卷号：正体 / 粗体
- 期号：保留 / 省略
- 页码：首页 / 首末页 / 页码范围连接符（hyphen vs en-dash）
- DOI：必需 / 可选 / 不要
- 作者截断规则：列全 / 若干作者后 et al.
- 作者-标题分隔符：句号 / 冒号 / 其他
- 结尾标点：句号 / 无

输出要求：
- 列出"**参考文献格式字段核对表**"
- 对每个字段标注：
  - 已确认
  - 缺失
  - 冲突
  - 需作者确认

---

**2.3 统计报告格式**

- *P* 值：统一为期刊要求格式（如斜体 *P*、精确值除非 <0.001）
- CI：统一为期刊要求格式（如 `[1.04, 1.15]`、`(1.04–1.15)` 等）
- OR / HR / RR：统一表述
- 效应量、CI、*P* 值在正文、表格、图注、补充材料之间统一
- 若稿件存在"raw P / adjusted P / Holm-adjusted P / FDR-adjusted q"等多种写法，必须统一并明确其含义
- SD 表示方式（如 `± SD` vs `(standard deviation X.X)`）按期刊要求统一

---

**2.4 缩写**

- 摘要中的缩写：按期刊要求精简，只保留核心术语
- 正文中首次出现时展开全称
- 缩写表（如期刊要求）必须补齐
- UK / US English 的缩写和拼写体系要统一

---

**2.5 关键词**

- 核查数量、排序、拼写规范
- 核查是否需采用 MeSH / 主题词 / 首字母排序
- 若期刊有限制（如避免缩写、避免句号），按其要求统一

---

**2.6 声明章节补齐**

- 检查期刊要求的每一个声明章节是否存在
- 缺失的用占位符标注：`[USER: INSERT ...]`
- AI 使用声明：如使用了 AI 工具，按期刊要求格式声明
- 不替用户虚构 Funding、COI、Author contributions、ORCID、Corresponding author 等信息

**2.6.1 代码仓库与 License**

- 检查 Data/Code availability、Supplementary notes 或投稿包说明中是否存在 `代码仓库 URL` / `License` 占位
- 若用户明确允许调用 GitHub MCP / skills，则只做最小必要的代码公开准备：
  - 确认或创建目标仓库
  - 整理最小可复现代码子集
  - 补齐 `LICENSE`
  - 回填最终仓库 URL 与 License 名称
- 若仓库尚不适合公开（如含敏感数据、机构限制、依赖未清理、公开范围未获确认），不得强推公开；保留占位并列出阻塞项

---

**2.7 表格与数字一致性**

- 确保所有表格的数据完整、无占位符
- 核查正文中的数字与表格中的数字一致
- 核查图注、补充材料、Cover Letter 中提及的关键数字与主稿一致
- 若存在"正文已更新但表格未同步"或"图注仍为旧值"，必须先修复再进入 Phase 3

输出要求：
- 逐项列出"**Markdown 预处理修改清单**"
- 每项标注：
  - 已完成
  - 待完成
  - 阻塞
  - 是否需要回退主干

**Phase 3：Word 文档生成**

---

**3.1 文档生成工具**

允许使用以下任一方案生成 `.docx`：

- npm `docx`
- `python-docx`
- `pandoc`

工具选择原则：
- 优先选择当前环境中最稳定、最可验证的实现
- 生成工具本身不是合规依据，验证结果才是依据
- 在统一验证脚本通过前，不得声明"已完成"

若使用 npm 方案，可按需执行：

```bash
cd [submission_package_dir]
npm init -y
npm install docx
```

若使用 Python 方案，则记录所用包与版本。

---

**3.2 文档结构要求**

| 组件 | 格式规范 |
|------|---------|
| 字体 | Times New Roman 12pt（或期刊指定） |
| 行距 | 双倍行距（或期刊指定） |
| 页边距 | 1 英寸 / 2.54 cm（或期刊指定） |
| 页面 | A4 或 US Letter（按期刊） |
| 页码 | 按期刊要求位置 |
| 页眉 | 按期刊要求 |
| 标题 | 按期刊要求；若无明确要求，可用 H1/H2/H3 分层实现 |

**3.2.1 段落默认值**

每个段落必须**显式设置**（不依赖 python-docx / docx 库默认值）：

- `alignment`：明确指定 JUSTIFY / LEFT / CENTER（不允许 None）
- `font.color.rgb = RGBColor(0, 0, 0)`：强制黑色（不依赖默认）
- `font.name = 'Times New Roman'`：强制
- `font.size = Pt(12)`：强制
- `paragraph_format.line_spacing`：强制双倍或期刊指定

每个 run 必须**显式设置**：

- `font.color.rgb = BLACK`（即使父段落已设置）
- `font.name` 和 `size` 不允许为 None

理由：python-docx 的"None"默认值在 Word 中可能渲染为不一致的颜色或字体；占位符红色也必须显式改回黑色才能投稿。

段落对齐默认值（除非期刊另有要求）：
- 正文段落：**两端对齐 (JUSTIFY)**
- 标题：左对齐
- 参考文献（悬挂缩进）：左对齐
- 表格数据列：居中；表格文本列：左对齐
- 图注：左对齐或按期刊示例

---

**3.3 表格格式：三线表**

医学期刊标准三线表规范：

- 表头上方：粗线
- 表头下方：细线
- 表尾下方：细线
- 无竖线
- 无底纹
- 表注在表尾下方，字号小一号，斜体或按期刊要求

**3.3.1 三线表结构验证**

生成三线表后，必须进行结构级验证，不得仅凭视觉判断。

最低验证要求：
- 表头上方为粗线等价设置
- 表头下方为细线等价设置
- 表尾下方为细线等价设置
- 不存在左/右竖线
- 不存在内部横竖网格线
- 不存在底纹或单元格 shading

若使用支持 XML/结构读取的工具，应优先做结构审计；
若当前工具链无法直接读取 XML，则必须至少通过导出检查与可视复核双重确认。

输出要求：
- 对每张表标记：
  - `三线表结构通过`
  - `三线表结构未通过`
  - `需人工复核`

---

**3.4 结构化摘要**

- 每个标签（如 Background / Methods / Results / Conclusions）加粗斜体或按期刊示例
- 标签后接冒号与正文
- 若期刊明确要求非结构式摘要，不得擅自结构化

---

**3.5 参考文献**

- 悬挂缩进
- 按期刊要求设置行距
- 引用样式必须与 Phase 1 / 1.5 确认结果一致
- 引用在标点前还是后，必须执行到位
- 不允许只把正文转换成 docx 而忽略参考文献的版式细节
- 期刊名特殊样式（如 Anesthesiology 要求 small caps）必须用 XML 级别样式实现，不能仅靠 Markdown 字符替换

---

**3.6 图注**

- 图号加粗，后接正文
- 每条图注之间留间距
- 图注中的统计量、缩写、样本量与主稿一致
- 若期刊要求图注另页、单独分页或统一置后，必须执行

---

**3.7 上标引用 / 方括号引用**

- 将 Markdown 中的引用统一转换为目标期刊要求的 Word 格式
- 不得保留 Markdown 风格的半成品引用
- 若期刊要求方括号数字，则严格按方括号数字执行
- 若期刊要求上标，则使用 Word superScript 格式实现

---

**3.8 各章节分页**

- 按期刊要求在主要章节之间插入分页符
- 若期刊要求表格和图注置后，则按规定另起页
- 不得在分页规则未确认前凭习惯处理

---

**3.9 构建后验证**

如环境提供验证脚本，应执行统一验证，例如：

```bash
python scripts/office/validate.py output.docx
```

若没有现成脚本，则验证脚本必须报告以下**全部项目**：

**结构层：**
- 文件是否成功生成
- 文档段落数
- 表格数

**渲染层：**
- **颜色分布**：黑色 run 数 / 红色 run 数 / 默认色 run 数（红色应为 0）
- **对齐分布**：JUSTIFY / LEFT / CENTER / 默认 的段落数
- **字体一致性**：所有 run 的 `font.name` 是否统一为期刊要求
- **字号分布**：所有 run 的 `font.size` 直方图（应只有 14/13/12/11 pt 几个值）
- **上标 run 数**（应等于参考文献引用次数）
- **占位符数**（`[USER:INSERT]` 出现次数和位置）
- **三线表结构审计**（每张表）

**边距与页面设置层：**
- 页边距 / 字体 / 行距抽样检查结果
- 引用样式抽样检查结果
- 页码位置确认

**任一项不符合期刊要求 → ❌ 阻塞，必须修复后重新生成。**

---

**3.10 清理构建产物**

若使用临时构建环境，构建完成后清理不应提交的中间产物，例如：

```bash
rm -rf node_modules package.json package-lock.json build_docx.js
```

必须额外验证清理结果，不得只写"已清理"。

输出要求：
- 给出"**Word 生成结果**"，至少包括：
  - 生成的 `.docx` 文件路径
  - 文件大小
  - 段落数
  - 表格数
  - 验证结果（按 3.9 完整清单）
  - 是否存在需人工复核的问题

**Phase 4：投稿包整理**

---

**4.1 目录结构**

推荐目录结构：

```text
submission_package/
├── manuscript_[journal].docx
├── manuscript_draft.md
├── cover_letter.md
├── figures/
│   ├── Figure1_xxx.pdf/.png
│   ├── Figure2_xxx.pdf/.png
│   └── ...
├── tables/
│   ├── Table1_xxx.csv (或 .docx)
│   └── Table2_xxx.csv (或 .docx)
├── supplement/
│   ├── eTable1_xxx.csv
│   ├── eFigure1_xxx.pdf/.png
│   └── ...
└── checklist/
    └── STROBE_checklist.md
```

若目标期刊有特定命名要求（如 Anesthesiology 要求 SDC 命名），以期刊要求为准。

---

**4.2 文件命名规范**

- 无空格，用下划线
- Figure 编号与正文一致
- Supplementary / eTable / eFigure / SDC 命名与正文引用一致
- 文件名不允许混用旧编号和新编号
- 图表文件必须可被编辑部一眼定位，不使用含糊命名

---

**4.3 Cover Letter 核查**

- 与最终版主稿的 Title、主要发现、定位完全一致
- 不含已删除的旧主张
- 不含过强术语
- 包含 COI 声明（如期刊要求）

必须额外执行一致性核查：
- Cover Letter 中出现的 AUC / OR / HR / RR / *P* 值，必须全部能在主稿中找到对应来源
- Cover Letter 中的 Title，必须逐字与主稿 Title 匹配
- Cover Letter 中的关键术语（模型名、评分名、亚组名、终点名）必须与主稿一致
- 若发现旧数字、旧术语或旧结论残留，必须标红并回到本阶段修复

---

**4.4 投稿包最小文件集**

若目标期刊未另行规定，至少核查以下文件是否齐全：

- 主稿 `.docx`
- 主稿 Markdown 源文件
- Cover Letter
- Figures
- Tables / Supplementary Tables
- Supplementary appendix / Additional file
- Checklist（如 STROBE / PRISMA / TRIPOD / CONSORT 等）
- Title page（若独立）
- Declarations / COI / AI statement（若独立）
- 代码仓库 URL / License 信息（若稿件要求）

任一硬性文件缺失，必须在最终判断中列为阻塞项。

输出要求：
- 给出"**投稿包文件清单**"，标明：
  - 文件名
  - 是否存在
  - 是否已核对命名
  - 是否与正文引用一致
  - 是否仍需用户补充

**Phase 4.5：投稿系统准备清单**

按目标期刊投稿系统（Editorial Manager / ScholarOne / 同类系统）的操作要求：

| 项 | 通常要求 | 状态 |
|----|---------|------|
| 主稿 | 单一 Word 文件含 Title + Abstract + Body + Refs + Figure Legends | ✅ / ⚠️ |
| Tables | 每张表独立 Word 文件（不接受 CSV/Excel） | ⚠️ 需用户转换 |
| Figures | 每图独立文件 (TIFF/EPS/PDF, 期刊要求的 dpi) | ⚠️ 当前格式与 dpi 检查 |
| SDC | 独立 PDF 文件，每个含独立标题/图注 | ⚠️ 命名规范 |
| Cover Letter | 系统内文本框或独立文件 | ✅ |
| Author info | 系统内逐个录入（不依赖稿件文本） | 由用户操作 |
| COI | 系统内每位作者独立填写 | 由用户操作 |
| Suggested reviewers | 期刊建议提供 | ⚠️ 用户自选 |
| Cover letter 中的 AI 声明 | 部分期刊要求 | ⚠️ 检查 |

明确标注每项是"投稿包内已就绪"还是"投稿系统内用户操作"。

**Phase 5 通过标准**

**🔴 红色（绝对阻塞，必须本 prompt 内解决）：**
- 字数超限（项 1, 2, 3）
- 必需声明缺失（项 6）
- 数字不一致（项 13, 15a）
- 边界违规（应在 B6 处理但若残留必须回退）
- DOCX 生成失败（项 16）
- 三线表结构未通过（项 9, 9a）
- 颜色非全黑（项 16b）
- 对齐方式错误（项 16a）

**🟡 黄色（用户必须解决，但不阻塞 prompt 完成）：**
- `[USER:INSERT]` 占位符（明确告知用户填写位置）
- 期刊要求但分析未做（如 sex-stratified analysis）→ 应在 Limitation 中说明
- 部分参考文献 DOI 缺失（标记 `[USER: INSERT DOI]` 占位）
- 投稿系统层任务（如 Tables 转独立 Word 文件、Figures 高 dpi 转换）

**🟢 绿色（推荐但非强制）：**
- 多语言润色
- 高 dpi 图（投稿后修改要求时再处理）
- 高级图表样式优化

**判定规则：**
- 任一 🔴 项 = ❌ → 不允许声明"已达到可提交状态"
- 多项 🟡 项 = ⚠️ → 允许声明，但须列明用户后续动作清单
- 全部 🟢 → 不影响判定

**最终输出格式**

1. Phase 0 输出：用户阻塞项前置确认清单
2. Phase 1 输出：期刊要求 vs 当前状态对照表
3. Phase 1.5 输出：官方指南 vs 已发表文章差异表
4. Phase 2 输出：Markdown 预处理修改清单（按 P0/P1/P2 分级）
5. Phase 3 输出：Word 生成结果（文件路径、大小、段落数、渲染验证状态）
6. Phase 4 输出：投稿包文件清单
7. Phase 4.5 输出：投稿系统准备清单
8. Phase 5 输出：最终验证清单（每项标 🔴 / 🟡 / 🟢 / ✅）
9. 最终判断：投稿包是否已达到可提交状态
10. 若未达到：明确列出阻塞项、对应 Phase、以及是否需要回退到 `academic-revision-system`

**最终判断规则**

只有在以下条件全部满足时，才可输出"**已达到可提交状态**"：

- Phase 0 已完成（所有用户阻塞项已确认）
- Phase 1 已通过，且无官方格式信息缺口
- Phase 1.5 已完成，且无未解决的指南冲突
- Phase 2–4.5 所有执行项已落地
- Phase 5 达到通过标准（无 🔴 阻塞，🟡 已明确列出用户后续动作）
- 不存在需要回退主干处理的硬伤

若任一条件不满足，必须明确输出：
- 当前状态：未达到可提交状态
- 阻塞项列表
- 每个阻塞项对应的 Phase
- 若需回退主干，应明确回到哪一步

**回退到 academic-revision-system 的明确触发条件**

**回退（必须）：**
- 发现某个数字与源数据不一致 → 回 B7（方法-结果一致性）
- 发现结论越界（如新增"recommended for clinical use"） → 回 B6
- 发现统计方法描述错误 → 回 B8
- 期刊要求的某个声明无法补齐（如 IRB 号缺失） → 阻塞，请求用户提供后回 B9
- 发现某个图与正文叙事冲突 → 回 B10

**不回退（本 prompt 内解决）：**
- Word count 超限 → 本 prompt 内削减
- 参考文献格式 → 本 prompt 内修
- 字体/对齐/颜色 → 本 prompt 内修
- 占位符标注 → 本 prompt 内做
- DOCX 渲染问题 → 本 prompt 内重新生成
