# PROMPT_GOVERNANCE

> 本文件是 prompt 体系的最小架构契约。
> 任何结构变更、新增 prompt、分组调整、pack 上线或回退，都应先满足本文件约束，再进入实现或落盘。
> 若某次局部优化与本契约冲突，以本契约为准；不要因为单次看起来更顺手，就破坏既有分层。

## 1. 分层原则

当前 prompt 体系固定为四层：

- **入口层**：`academic-revision-quick-nav`
- **主干层**：`academic-revision-system`
- **落地层**：`submission-word-package`
- **窄任务层**：各类 pack（如 `reviewer-comment-triage`、审计工具）

四层职责必须互不抢活：

- `quick-nav` 只做入口，不承载完整 prompt 正文
- `system` 只做主流程与专项正文
- `submission` 只做投稿包、格式化、终检与投稿物料落地，不替代主干，也不承接高重叠审计 pack
- `pack` 只做窄任务，不替代主干步骤

## 2. 入口层约束

`academic-revision-quick-nav` 只允许存在入口条目。

每条入口严格封顶三要素：

- 用途
- 对应
- 触发语

除标题外，任一入口不得超过上述三要素。
若条目重新出现输出模板、表格骨架、完整 prompt 正文，视为向模板层回流，必须回退。

`常用工具(转发入口)` 保留在入口层，**不得并回** `03 专项模块`。

## 3. 主干层约束

`academic-revision-system` 是唯一活跃主干。
所有完整流程步骤、研究画像、铁律、专项正文，都应以它为准。

当前主干步骤固定为：

- 写作前阶段：`A0–A7`
- 修稿投稿阶段：`B0–B13`

如需新增主干步骤，必须先论证：

1. 该问题无法由现有步骤吸收
2. 该步骤不是 pack 可承载的窄任务
3. 新编号不会破坏现有调用链

## 4. 落地层约束

`submission-word-package` 承担投稿包生成、格式化、终检与投稿物料落地。

落地层不得：

- 承接主干职责（主线诊断、修改总纲、步骤级审查）
- 合并高重叠审计 pack 的内容
- 被任何 pack 取代或架空

主干与落地层的边界：主干 B11（投稿前终检）做判断性终检；`submission` 做执行性落地。判断不得越界执行，执行不得越界判断。

## 5. pack 准入规则

pack 只用于承载**高频、低重叠、边界清晰的窄任务**。

当前正式轨道 pack 只有 4 个：

- `reviewer-comment-triage`
- `reference-integrity-audit`
- `figure-table-citation-audit`
- `statistical-reporting-harmonizer`

任何候选 pack 在进入正式轨道前，必须先回答：

1. 它为什么不是 `system` 的增强迭代？
2. 它为什么不是 `submission` 的增强迭代？
3. 它是否会让用户再次面对"同一件事两个入口"？
4. 它是否能用一句话说清自己的边界？

若上述任一问题无法明确回答，默认**不得进入正式轨道**。

## 6. `_hold` 规则

与 `system` 或 `submission` 高重叠的候选项，默认进入 `_hold`，不得直接纳入正式分组。

进入 `_hold` 不等于删除，也不等于否定内容质量，只表示：

- 当前职责位置未定
- 需要等待真实使用信号
- 暂不计入正式体系盘点

## 7. parser 兼容规则

步骤内部的输出模板、小节或表格说明，不得使用会被 parser 识别成独立 prompt 条目的标题层级。

若 parser 规则为：

- `#` = 分组
- `##` = 条目

则步骤内部模板必须使用 `###` 或更低层级。

任何导致运行时菜单暴增的标题层级写法，都视为结构性错误，优先修复。

## 8. 联动变更顺序规则

当一次变更同时涉及主干、legacy、路由三者联动时，默认顺序为：

1. 先修主干
2. 再清 legacy
3. 最后调路由

pack 只在主干稳定后新增；新增后须先试跑，再决定是否升级为 v1。

单独的 pack 热修、parser 兼容修复、quick-nav 单条触发语调整等日常维护，不受本条顺序约束。

## 9. 当前阶段判断

当前体系已进入**可维护期**，不是继续大规模重构期。

当前首要任务是：

- 守住边界
- 控制增量
- 让正式轨道 pack 经真实试跑自然分层

不是继续扩张分组、合并层级或新增第二批 pack。

## 10. 治理文件位置规则

本文件及任何未来同级治理文件，不得放入 `prompts/` 运行时目录，不得被 parser 扫描，不得出现在任何 UI 入口列表中。

治理文件的正确位置是**项目根**或 `docs/`，与被治理的 prompt 体系在目录层级上明确分离。

不得依赖 `favorite: false` 或类似软性标记来"让治理文件在运行时目录中隐身"——位置隔离优先于属性隔离。

每条事件对应一条或多条约束，非完整事件史。

- **2026-04: quick-nav 路由化**
  `academic-revision-quick-nav` 曾长期作为"简化版平行主系统"，复刻主干模板。路由化时收敛为三要素入口。此事件确立了入口层约束（第 2 条）。

- **2026-04: 阶段二误纳入事件**
  `claim-evidence-display-audit` 和 `submission-readiness-audit` 曾因"内容写得不错"被默认追认为正式轨道 pack，后发现与 `submission-word-package` 的结果-图表承载矩阵、最终验证清单高度重叠。回退至 `_hold`。此事件确立了 pack 准入规则和 `_hold` 规则（第 5、6 条）。

- **2026-04: parser 误识别事件**
  `academic-revision-system.md` 中步骤内部模板使用 `##` 二级标题，被 parser 识别为独立 prompt 条目，导致运行时 `02 修稿投稿阶段` 出现 57 项而非 14 项。批量降级为 `###` 后修复。此事件确立了 parser 兼容规则（第 7 条）。

- **2026-04: 入口层误合并提议**
  曾出现"将 `常用工具(转发入口)` 并回 `03 专项模块`"的建议。该建议在 UI 清爽度上合理，但会把入口层组件搬入主干层，违反四层分工。拒绝采纳。此事件确立了入口层约束中的显式反例记录（第 2 条末句）。

- **2026-04: 治理文件位置错误提议**
  治理文件起草阶段，曾出现"将 `PROMPT_GOVERNANCE.md` 放入 `prompts/` 根目录，靠 `favorite: false` 降级"的建议。该建议会让治理文件被 parser 扫描，从而从"硬约束"降格为"另一个 prompt"，威胁契约自身的权威性。拒绝采纳。此事件确立了治理文件位置规则（第 10 条）。
